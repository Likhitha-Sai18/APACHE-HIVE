# Hive Bucketing — Complete Concept (Easy + Deep Understanding)

Bucketing is one of the most important optimization techniques in Hive.

Most students understand **partitioning**, but get confused about **when bucketing is needed** and **how Hive actually stores data**.

Let's build the concept from scratch.

---

# 1. Why Do We Need Bucketing?

Suppose you have a hospital dataset:

| Patient_ID | Department | Severity |
| ---------- | ---------- | -------- |
| 101        | Cardiology | Minor    |
| 102        | Oncology   | Major    |
| 103        | Cardiology | Minor    |
| 104        | Neurology  | Critical |
| ...        | ...        | ...      |

Imagine:

* 100 Million Records
* Patient IDs are unique
* Millions of different Patient IDs

---

## Can We Partition by Patient_ID?

Example:

```sql
PARTITIONED BY (patient_id)
```

Hive will create:

```text
patient_id=101/
patient_id=102/
patient_id=103/
patient_id=104/
...
patient_id=100000000/
```

Millions of directories!

### Problem

The NameNode must maintain metadata for every directory.

Result:

❌ Huge metadata overhead

❌ Slower query planning

❌ Too many small files

❌ Storage inefficiency

---

# Solution: Bucketing

Instead of creating millions of partitions,

Hive creates a fixed number of files called buckets.

Example:

```sql
CLUSTERED BY (patient_id)
INTO 10 BUCKETS
```

Now Hive creates only:

```text
bucket_00000
bucket_00001
bucket_00002
...
bucket_00009
```

Only 10 files.

Much more efficient.

---

# Simple Definition

### Partitioning

Creates separate directories.

### Bucketing

Creates separate files inside a directory.

---

# How Does Hive Decide Which Bucket Gets a Record?

Hive uses:

```text
hash(bucket_column) % number_of_buckets
```

Formula:

```text
Bucket Number =
hash(value) % N
```

Where:

* value = bucket column value
* N = number of buckets

---

# Example

Suppose:

```sql
CLUSTERED BY(patient_id)
INTO 10 BUCKETS
```

Patient ID:

```text
101
```

Hive computes:

```text
hash(101)=23451
```

Then:

```text
23451 % 10 = 1
```

Goes to:

```text
bucket_1
```

---

Patient ID:

```text
102
```

```text
hash(102)=45672
```

```text
45672 % 10 = 2
```

Goes to:

```text
bucket_2
```

---

Patient ID:

```text
103
```

```text
hash(103)=78903
```

```text
78903 % 10 = 3
```

Goes to:

```text
bucket_3
```

---

# Important Rule

Records with the same bucket key always go to the same bucket.

Example:

Patient_ID = 101

```text
Record A → bucket_1
Record B → bucket_1
Record C → bucket_1
```

This makes searching faster.

---

# Why Searching Becomes Faster?

Suppose query:

```sql
SELECT *
FROM hospital
WHERE patient_id = 101;
```

Without Bucketing:

```text
File1
File2
File3
File4
File5
...
```

Hive scans all files.

---

With Bucketing:

Hive knows:

```text
101 → bucket_1
```

It scans only:

```text
bucket_1
```

instead of all buckets.

Huge performance gain.

# Partition + Bucketing

Most common in industry.

```sql

    department STRING,

)
CLUSTERED BY (patient_id)
INTO 10 BUCKETS;
```

Your instructor mentioned:

Because:

Each reducer creates one bucket file.

```text
5 Reducers
```

```text

```

# How Many Buckets Should We Create?

No fixed rule.

---

### Practical Formula

```text
Number of Buckets
≈
Total Data Size
/
HDFS Block Size
```

Example:

```text
Data = 256 MB
Block Size = 128 MB
```

```text
256 / 128 = 2
```

Use:

```sql
INTO 2 BUCKETS
```

# What is Data Skew?

Suppose:

```text
bucket_1 = 95% records
bucket_2 = 1%
bucket_3 = 1%
bucket_4 = 3%
```

This is skewness.

Bad distribution.

---

Good Bucketing:

```text
bucket_1 = 25%
bucket_2 = 25%
bucket_3 = 25%
bucket_4 = 25%
```

Balanced workload.

# When Should We Use Bucketing?

Use Bucketing When:

✅ High cardinality columns

Examples:

```text
Patient_ID
Customer_ID
Transaction_ID
Order_ID
Employee_ID
```

Millions of unique values.

---

Avoid Partitioning On:

```text
Customer_ID
Patient_ID
Order_ID
```

because too many partitions.

Use Bucketing instead.

# Partitioning vs Bucketing

| Feature           | Partitioning            | Bucketing                |
| ----------------- | ----------------------- | ------------------------ |
| Storage           | Directories             | Files                    |
| Best For          | Low Cardinality Columns | High Cardinality Columns |
| Example           | Gender, State           | Customer_ID              |
| Data Pruning      | Yes                     | Partial                  |
| Query Speed       | Very Fast               | Fast                     |
| Metadata Overhead | High                    | Low                      |
| Number Created    | Dynamic                 | Fixed                    |

# Real Interview Question

### Why not partition on Patient_ID?

**Answer:**

Patient_ID has very high cardinality (millions of distinct values). Partitioning would create millions of directories causing NameNode metadata overhead. Bucketing stores records into a fixed number of bucket files, making storage and query execution more efficient.

---

# One-Line Revision

### Partitioning

**Splits data into folders (directories).**

### Bucketing

**Splits data into files (buckets) using Hash Function.**

### Formula

```text
Bucket Number =
hash(column_value)
%
number_of_buckets
```

### Industry Rule

**Partition on low-cardinality columns (State, Department, Date).**

**Bucket on high-cardinality columns (Customer_ID, Patient_ID, Transaction_ID).**

This "Partition first → Bucket inside partitions" pattern is the most common Hive design used in real-world Data Engineering projects.
