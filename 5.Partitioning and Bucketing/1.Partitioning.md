# Understanding Hive Partitioning from Scratch

Before understanding partitioning, you must understand the problem it solves.

Imagine a hospital has:

```text
100 Million Patient Records
```

stored in a Hive table called:

```sql
patients
```

Data:

| Patient_ID | Name  | Department |
| ---------- | ----- | ---------- |
| 1          | John  | ENT        |
| 2          | Alice | Gynecology |
| 3          | Bob   | Cardiology |
| 4          | Emma  | Pediatrics |
| ...        | ...   | ...        |

# The Problem Without Partitioning

Suppose a doctor asks:

> Show all ENT patients.

Query:

```sql
SELECT *
FROM patients
WHERE department='ENT';
```

Most beginners think Hive directly finds ENT records.

Actually it doesn't.

Hive sees:

```text
patients table
```

but has no idea where ENT records are physically stored.

The records are mixed together.

```text
ENT
Gynecology
Cardiology
ENT
Pediatrics
ENT
Cardiology
...
```

Therefore Hive must:

```text
Open file
Read row 1
Check department

Read row 2
Check department

Read row 3
Check department

...
```

until the last record.

This is called:

# Full Table Scan

```text
Entire Table Read
```

Even if:

```text
100 Million Records
```

exist and only:

```text
1000 Records
```

belong to ENT,

Hive still scans:

```text
100 Million Records
```

This is expensive.

---

# Why Full Table Scan Is Bad

Suppose:

```text
1 TB Dataset
```

Query needs:

```text
10 MB Data
```

Without partitioning:

```text
Read 1 TB
Use 10 MB
```

Waste of:

```text
Disk I/O
CPU
Memory
Network
Time
```

Big Data systems cannot afford this.

This is exactly why Partitioning was introduced.

---

# The Main Idea Behind Partitioning

Partitioning means:

> Physically dividing table data into separate folders based on a column value.

Instead of:

```text
patients

└── datafile
```

Hive creates:

```text
patients

├── department=ENT

├── department=Gynecology

├── department=Cardiology

└── department=Pediatrics
```

Now records are separated.

---

# What Happens Internally

ENT records:

```text
1,John,ENT
10,Rahul,ENT
20,David,ENT
```

go into:

```text
department=ENT
```

folder.

Gynecology records:

```text
2,Alice,Gynecology
15,Sarah,Gynecology
```

go into:

```text
department=Gynecology
```

folder.

---

# How Query Processing Changes

Same query:

```sql
SELECT *
FROM patients
WHERE department='ENT';
```

Hive now sees:

```text
department='ENT'
```

and immediately knows:

```text
ENT records are inside

department=ENT
```

directory.

Therefore Hive:

```text
Skips Gynecology
Skips Cardiology
Skips Pediatrics
```

and reads only:

```text
department=ENT
```

This is called:

# Partition Pruning

---

# Partition Pruning (Most Important Concept)

Interviewers love this question.

Definition:

> Hive automatically eliminates irrelevant partitions and reads only the required partition.

Example:

Query:

```sql
SELECT *
FROM patients
WHERE department='ENT';
```

Partitions:

```text
ENT
Gynecology
Cardiology
Pediatrics
```

Hive prunes:

```text
Gynecology
Cardiology
Pediatrics
```

and reads:

```text
ENT
```

only.

---

# Why WHERE Clause Is Important

Partitioning helps only when the WHERE condition uses the partition column.

Partition column:

```text
department
```

Good Query:

```sql
SELECT *
FROM patients
WHERE department='ENT';
```

Pruning happens.

---

Bad Query:

```sql
SELECT *
FROM patients
WHERE patient_name='John';
```

Why?

Because:

```text
patient_name
```

is not a partition column.

Hive cannot identify a partition.

Therefore:

```text
All partitions scanned.
```

Now let's discuss the first type.

---

## Concept

Hive creates:

```text

     ↓

severity=Extreme
```

directory.

Then data is inserted.

---

Result:

```text

└── department=Anesthesia
       │
       └── severity=Extreme
```

---

# Why Static Partitioning Exists

Suppose source file:

```text

```

contains no country column.

But business requirement:

```text

```

Since:

```text

```

is not present in data,

Hive cannot dynamically determine it.

Static partition becomes useful.

Additional processing required.

in speed.

# Interview Summary

If an interviewer asks:

### What is Hive Partitioning?

Answer:

> Hive Partitioning is a data organization technique in which table data is physically divided into separate directories based on partition column values. When queries filter on partition columns, Hive uses Partition Pruning to scan only the relevant partitions, reducing data scans and significantly improving performance.

### Difference Between Static and Dynamic Partitioning?

Answer:

> In Static Partitioning, partition values are explicitly specified by the user during data insertion. In Dynamic Partitioning, Hive automatically determines partition values from incoming data and creates partitions automatically.

# 4. Partitioning vs Bucketing

This is one of the most asked Hive interview questions.

---

# Why Bucketing Was Introduced

Partitioning works well when values are limited.

Good Example:

```text
Department

ENT
Gynecology
Pediatrics
```

Only a few folders.

---

Bad Example:

```text
Customer_ID
```

Values:

```text
1
2
3
...
10 Million
```

Partitioning creates:

```text
customer_id=1

customer_id=2

customer_id=3

...
```

10 million directories.

Terrible.

---

# Bucketing Solution

Instead of creating folders for each value:

Hive creates:

```text
bucket_1

bucket_2

bucket_3

bucket_4
```

Fixed number.

---

# How Bucketing Works

Hive uses:

```text
Hash Function
```

Formula:

```text
hash(column) % number_of_buckets
```

Example:

```text
Customer_ID = 103

hash(103)%4 = 3
```

Record goes to:

```text
bucket_3
```

---

# Bucketed Table Example

```sql
CREATE TABLE customers
(
 customer_id INT,
 customer_name STRING
)

CLUSTERED BY(customer_id)

INTO 4 BUCKETS;
```

---

# Storage Structure

```text
customers

bucket_00000

bucket_00001

bucket_00002

bucket_00003
```

---

# Comparison Table

| Feature        | Partitioning    | Bucketing        |
| -------------- | --------------- | ---------------- |
| Based On       | Column Values   | Hash Function    |
| Storage        | Directories     | Files            |
| Number Created | Variable        | Fixed            |
| Best For       | Low Cardinality | High Cardinality |
| Example        | Country         | Customer_ID      |

---

# When To Use Which?

Partitioning:

```text

```

Bucketing:

```text

```

```text

```

