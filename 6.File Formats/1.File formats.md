# Hive ORC File Format – Complete Conceptual Notes

## 1. Why Do We Need Different File Formats in Hive?

By default, Hive stores data as **Text Files**.

Example:

| ID | Name  | Salary |
| -- | ----- | ------ |
| 1  | Ram   | 50000  |
| 2  | Ravi  | 60000  |
| 3  | Priya | 70000  |

Stored as text:

```text
1,Ram,50000
2,Ravi,60000
3,Priya,70000
```

### Problem with Text Files

* Large storage size
* Slow query performance
* No efficient compression
* Entire file must be scanned

As data grows to TBs or PBs, text files become inefficient.

Hence Hive supports optimized file formats:

1. ORC
2. Parquet
3. Sequence File
4. Avro
5. RCFile

Among them, **ORC and Parquet are most widely used**.

---

# ORC (Optimized Row Columnar)

ORC is a highly optimized storage format developed for Hive.

### Definition

ORC stores data:

✅ Column-wise

✅ Compressed

✅ Indexed

✅ Optimized for Hive queries

---

# Columnar Storage Concept

Suppose we have:

| ID | Name  | Salary |
| -- | ----- | ------ |
| 1  | Ram   | 50000  |
| 2  | Ravi  | 60000  |
| 3  | Priya | 70000  |

---

## Traditional Row Storage

Stored as:

```text
1,Ram,50000
2,Ravi,60000
3,Priya,70000
```

Disk layout:

```text
Row1 → 1 Ram 50000
Row2 → 2 Ravi 60000
Row3 → 3 Priya 70000
```

Entire row stays together.

---

## ORC Columnar Storage

Stored as:

```text
ID Column
---------
1
2
3

Name Column
-----------
Ram
Ravi
Priya

Salary Column
-------------
50000
60000
70000
```

Disk layout:

```text
ID      → 1 2 3

NAME    → Ram Ravi Priya

SALARY  → 50000 60000 70000
```

---

## Why is this Faster?

Suppose query:

```sql
SELECT salary
FROM employee;
```

Need only Salary column.

### Row Format

Reads:

```text
ID
Name
Salary
```

for every row.

---

### ORC Format

Reads only:

```text
Salary Column
```

No need to read:

```text
ID
Name
```

Result:

🚀 Less disk I/O

🚀 Faster query

🚀 Better performance

---

# ORC Internal Structure

Visual Structure:

```text
ORC File
│
├── Stripe 1
│    ├── Index Data
│    ├── Row Data
│    └── Stripe Footer
│
├── Stripe 2
│    ├── Index Data
│    ├── Row Data
│    └── Stripe Footer
│
├── Stripe 3
│    ├── Index Data
│    ├── Row Data
│    └── Stripe Footer
│
└── File Footer
```

---

## What is a Stripe?

A Stripe is a large chunk of rows.

Default size:

```text
256 MB
```

Can be changed.

Example:

```text
1000 Rows
```

may become

```text
Stripe 1 → Rows 1-300

Stripe 2 → Rows 301-600

Stripe 3 → Rows 601-1000
```

---

# Components of a Stripe

## 1. Index Data

Contains:

* Min value
* Max value
* Row offsets

Example:

```text
Salary Column

Min = 10000

Max = 50000
```

---

## Why Useful?

Query:

```sql
SELECT *
FROM employee
WHERE salary > 70000;
```

Stripe metadata:

```text
Stripe 1
Min=10000
Max=50000
```

Immediately skipped.

No need to read it.

---

### Stripe Skipping

```text
Stripe 1
Salary Max = 50000
❌ Skip

Stripe 2
Salary Max = 65000
❌ Skip

Stripe 3
Salary Max = 90000
✅ Read
```

This is called:

# Predicate Pushdown

One of the biggest ORC optimizations.

---

## 2. Row Data

Contains actual records.

Stored in:

```text
Columnar Format
```

Example:

```text
ID Column

Name Column

Salary Column
```

Actual business data resides here.

---

## 3. Stripe Footer

Contains metadata:

* Column information
* Encoding details
* Compression details
* Location of data

Acts like a mini catalog of the stripe.

---

# ORC Compression

ORC automatically compresses data.

Supported:

* ZLIB
* SNAPPY
* LZO

Example:

```text
Text File = 100 GB

ORC File = 20 GB
```

Possible 70–90% reduction.

---

# Advantages of ORC

## 1. Columnar Storage

Reads only required columns.

---

## 2. Compression

Less disk space.

---

## 3. Indexing

Stripe-level indexes.

---

## 4. Predicate Pushdown

Skips irrelevant stripes.

---

## 5. Faster Aggregations

Example:

```sql
SELECT SUM(salary)
FROM employee;
```

Reads only Salary column.

Very fast.

---

# ORC Example

Create ORC Table:

```sql
CREATE TABLE employee_orc(
id INT,
name STRING,
salary INT
)
STORED AS ORC;
```

---

Insert Data:

```sql
INSERT INTO employee_orc
SELECT *
FROM employee;
```

---

Check Format:

```sql
DESCRIBE FORMATTED employee_orc;
```

Output:

```text
InputFormat:
OrcInputFormat

OutputFormat:
OrcOutputFormat
```

# Parquet File Format

Another popular columnar format.

### Characteristics

✅ Columnar

✅ Compressed

✅ Distributed Analytics

✅ Spark Friendly

---

## Architecture

```text
Parquet
│
├── Row Groups
│
├── Column Chunks
│
└── Pages
```

---

# ORC vs Parquet

| Feature             | ORC       | Parquet          |
| ------------------- | --------- | ---------------- |
| Developed For       | Hive      | Hadoop Ecosystem |
| Compression         | Excellent | Excellent        |
| Query Speed in Hive | Faster    | Fast             |
| Spark Support       | Good      | Excellent        |
| Metadata            | Rich      | Rich             |
| Industry Usage      | Very High | Very High        |

---

## Industry Rule

### Mostly Hive

Use:

```text
ORC
```

### Mostly Spark

Use:

```text
Parquet
```

Though both are widely used.

# Sequence File

Older Hadoop storage format.

Stores:

```text
Key → Value
```

pairs.

Example:

```text
101 → Ram,50000

102 → Ravi,60000

103 → Priya,70000
```

---

# Storage Style

Unlike ORC:

```text
Sequence File
=
Row Storage
```

Stores:

```text
Entire Row Together
```

Example:

```text
101 Ram 50000

102 Ravi 60000

103 Priya 70000
```

---

# ORC vs Sequence File

| Feature             | ORC       | Sequence File |
| ------------------- | --------- | ------------- |
| Storage             | Columnar  | Row           |
| Compression         | Yes       | Yes           |
| Indexing            | Yes       | Limited       |
| Query Speed         | Very Fast | Moderate      |
| Analytics           | Excellent | Not Ideal     |
| Aggregation Queries | Excellent | Slower        |

---

# When to Use ORC?

Use ORC when:

✔ Hive Data Warehouse

✔ Analytical Queries

✔ Aggregations

✔ Large Datasets

✔ Partitioned Tables

✔ Frequent Reporting

Example:

```sql
SELECT state,
COUNT(*)
FROM covid_cases
GROUP BY state;
```

ORC is ideal.

# Interview Questions

### Q1. What is ORC?

Optimized Row Columnar storage format used in Hive for fast querying and compression.

---

### Q2. Is ORC row-based or column-based?

Column-based.

---

### Q3. What is a Stripe in ORC?

A large block of rows stored together, default size 256 MB.

---

### Q4. What does Index Data contain?

Minimum values, maximum values, and row offsets.

---

### Q5. What is Predicate Pushdown?

Skipping unnecessary stripes using metadata before reading data.

---

### Q6. Difference between ORC and Parquet?

Both are columnar formats. ORC is Hive-optimized, while Parquet is widely preferred in Spark ecosystems.

---

### Q7. Difference between ORC and Sequence File?

ORC → Columnar

Sequence File → Row-oriented key-value format.
