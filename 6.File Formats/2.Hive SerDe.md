# Hive SerDe (Serializer & Deserializer) – Complete Conceptual Notes

Before Hive can query data, it must understand:

* Where each column starts and ends
* How fields are separated
* What file format is being used
* How data should be converted into Hive rows and columns

This is exactly the job of **SerDe**.

---

# What is SerDe?

**SerDe = Serializer + Deserializer**

It is a component in Hive responsible for:

✅ Reading data from files

✅ Converting file data into Hive objects

✅ Converting Hive objects back into file format

Think of SerDe as a **translator** between:

```text
File Format
     ↕
   SerDe
     ↕
Hive Engine
```

---

# Why Do We Need SerDe?

Suppose you have a CSV file:

```csv
101,Ram,50000
102,Ravi,60000
103,Priya,70000
```

Hive sees:

```text
101,Ram,50000
```

But how does it know:

```text
101 = ID

Ram = Name

50000 = Salary
```

How does it know comma (,) is the separator?

Answer:

👉 SerDe tells Hive how to interpret the data.

# Two Parts of SerDe

## 1. Deserializer

Used while reading data.

```text
File
 ↓
Deserializer
 ↓
Hive Objects
```

---

## 2. Serializer

Used while writing data.

```text
Hive Objects
 ↓
Serializer
 ↓
File
```

---

# Deserializer (Read Operation)

Suppose file contains:

```csv
101,Ram,50000
```

Deserializer reads:

```text
Comma Separated Data
```

and converts it into:

```text
ID = 101

Name = Ram

Salary = 50000
```

Hive now understands the row.

---

## Visual Flow

```text
CSV File
│
├── 101,Ram,50000
├── 102,Ravi,60000
└── 103,Priya,70000

        ↓

Deserializer

        ↓

Hive Objects

Row 1:
ID=101
Name=Ram
Salary=50000

Row 2:
ID=102
Name=Ravi
Salary=60000
```

---

# Serializer (Write Operation)

Suppose Hive executes:

```sql
INSERT INTO employee
VALUES(104,'Kiran',70000);
```

Inside Hive:

```text
ID=104
Name=Kiran
Salary=70000
```

Serializer converts it into:

```csv
104,Kiran,70000
```

and writes it to HDFS.

---

## Visual Flow

```text
Hive Object

ID=104
Name=Kiran
Salary=70000

        ↓

Serializer

        ↓

104,Kiran,70000

        ↓

Stored in HDFS
```

```text

                WRITE

    ↓

    ↓
```

# Example with CSV File

CSV Data:

```csv
101,Ram,50000
102,Ravi,60000
```

Table:

```sql
CREATE TABLE employee(
id INT,
name STRING,
salary INT
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ',';
```

Hive uses:

```text
LazySimpleSerDe
```

to understand:

```text
Comma = Column Separator
```

---

# Example with JSON

File:

```json
{
"id":101,
"name":"Ram",
"salary":50000
}
```

CSV SerDe cannot read this.

Need:

```text
JSON SerDe
```

which understands:

```json
{
"key":"value"
}
```

structure.

---

# Example with ORC

ORC files are binary.

You cannot open them and read:

```text
101,Ram,50000
```

Instead:

```text
010101001001001
```

(binary encoded internally)

Hive needs:

```text
ORC SerDe
```

to decode the data.

---

# Example with Parquet

Parquet is also:

```text
Binary
+
Columnar
```

Hive uses:

```text
Parquet SerDe
```

to understand it.

# Common Hive SerDes

| File Type | SerDe Used       |
| --------- | ---------------- |
| CSV       | LazySimpleSerDe  |
| TSV       | LazySimpleSerDe  |
| JSON      | JsonSerDe        |
| ORC       | OrcSerde         |
| Parquet   | ParquetHiveSerDe |
| Avro      | AvroSerDe        |

---

# CSV SerDe Example

```sql
CREATE TABLE employee(
id INT,
name STRING,
salary INT
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ',';
```

Here Hive automatically uses:

```text
LazySimpleSerDe
```

---

# JSON SerDe Example

```sql
CREATE TABLE employee_json(
id INT,
name STRING,
salary INT
)
ROW FORMAT SERDE
'org.apache.hive.hcatalog.data.JsonSerDe';
```

Now Hive understands JSON.

---

# ORC SerDe Example

```sql
CREATE TABLE employee_orc(
id INT,
name STRING,
salary INT
)
STORED AS ORC;
```

Hive automatically uses:

```text
OrcSerde
```

No need to specify manually.

---

# Parquet SerDe Example

```sql
CREATE TABLE employee_parquet(
id INT,
name STRING,
salary INT
)
STORED AS PARQUET;
```

Hive automatically uses:

```text
ParquetHiveSerDe
```

---

# How Hive Chooses a SerDe?

When you create:

```sql
STORED AS ORC
```

Hive automatically selects:

```text
OrcSerde
```

When you create:

```sql
STORED AS PARQUET
```

Hive automatically selects:

```text
ParquetHiveSerDe
```

When you specify:

```sql
ROW FORMAT DELIMITED
```

Hive uses:

```text
LazySimpleSerDe
```

# Interview Questions

### Q1. What is SerDe in Hive?

SerDe stands for Serializer and Deserializer. It converts file data into Hive objects and vice versa.

---

### Q2. Why is SerDe required?

Because Hive must understand how data is stored and separated in different file formats.

---

### Q3. What does Deserializer do?

Reads file data and converts it into Hive-readable objects.

---

### Q4. What does Serializer do?

Converts Hive objects into file format before storing in HDFS.

---

### Q5. Which SerDe is used for CSV?

LazySimpleSerDe.

---

### Q6. Which SerDe is used for ORC?

OrcSerde.

---

### Q7. Which SerDe is used for Parquet?

ParquetHiveSerDe.

---

### Q8. Can Hive read JSON directly?

Yes, using JsonSerDe.
