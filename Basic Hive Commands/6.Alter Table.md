# ALTER TABLE Command in Hive

## Introduction

Once a table is created in Hive, its structure is not necessarily permanent. As business requirements change, the schema of a table may also need to change. New columns may need to be added, existing columns may need to be renamed, or even the table itself may require a new name.

Hive provides the **ALTER TABLE** command to perform such modifications without recreating the entire table.

The ALTER TABLE command is used for schema evolution, allowing changes to table metadata while preserving the existing data.

---

# What is Schema Evolution?

Schema evolution refers to modifying the structure of an existing table without deleting and recreating it.

Examples include:

* Renaming a table
* Adding columns
* Renaming columns
* Modifying column data types
* Replacing column definitions

This is especially useful in Big Data environments where tables may contain millions or billions of records. Recreating such tables can be time-consuming and expensive.

---

# ALTER TABLE Syntax

General syntax:

```sql
ALTER TABLE table_name
operation;
```

The operation can be:

* RENAME TO
* ADD COLUMNS
* CHANGE
* REPLACE COLUMNS

Each operation serves a different purpose.

---

# Renaming a Table

## Why Rename a Table?

A table name may need to be changed because:

* Naming conventions have changed.
* The table purpose has evolved.
* A typo was made during creation.
* A more meaningful name is required.

Example:

```text
temp_ext
      ↓
temp_exter
```

---

## Syntax

```sql
ALTER TABLE old_table_name
RENAME TO new_table_name;
```

---

## Example

```sql
ALTER TABLE temp_ext
RENAME TO temp_exter;
```

---

## Internal Working

When a table is renamed:

### Metadata Changes

Hive updates the table name in the Metastore.

```text
Before

temp_ext
```

```text
After

temp_exter
```

### Data Remains Unchanged

The actual table data is not modified.

### Schema Remains Unchanged

All columns and data types remain exactly the same.

Only the table identifier changes.

---

## Verification

Use:

```sql
SHOW TABLES;
```

to verify that the new table name appears in the database.

---

# Adding New Columns

## Why Add Columns?

As applications evolve, additional information may need to be stored.

Example:

Original Employee Table

| Name | Department |
| ---- | ---------- |
| John | IT         |

Later requirement:

| Name | Department | Company |
| ---- | ---------- | ------- |
| John | IT         | ABC Ltd |

A new column is needed.

---

## Syntax

```sql
ALTER TABLE table_name
ADD COLUMNS (
    column_name datatype
);
```

---

## Example

```sql
ALTER TABLE temp_exter
ADD COLUMNS (
    company STRING
);
```

---

## Result

Before:

| Name | Department |
| ---- | ---------- |

After:

| Name | Department | Company |
| ---- | ---------- | ------- |

---

## Internal Working

Hive updates the schema definition in the Metastore.

Existing records remain unchanged.

For previously existing rows:

```text
Company = NULL
```

because no value existed when those records were originally inserted.

---

## Important Point

Adding a column does not affect existing data.

Only the schema is extended.

---

# Renaming a Column

## Why Rename a Column?

Sometimes:

* Typographical mistakes occur.
* Naming standards change.
* More meaningful names are required.

Example:

```text
company
      ↓
firm
```

---

## Syntax

```sql
ALTER TABLE table_name
CHANGE old_column_name
new_column_name datatype;
```

---

## Example

```sql
ALTER TABLE temp_exter
CHANGE company firm STRING;
```

---

## Result

Before:

| company |
| ------- |

After:

| firm |
| ---- |

---

## Why Is Data Type Required?

Even if the data type is unchanged, Hive requires it during the CHANGE operation.

Example:

```sql
CHANGE company firm STRING
```

The STRING specification must still be provided.

---

## Internal Working

Hive updates:

```text
Column Name
```

inside the Metastore.

Actual data values remain unchanged.

Only the column metadata is modified.

---

# Replacing Columns

## Purpose

The REPLACE COLUMNS operation is more powerful than ADD or CHANGE.

It allows the complete column definition of a table to be replaced.

---

## Syntax

```sql
ALTER TABLE table_name
REPLACE COLUMNS (
    column1 datatype,
    column2 datatype,
    ...
);
```

---

## Concept

Instead of modifying one column at a time, Hive replaces the entire schema definition with the new list of columns.

Example:

Current Schema

```text
name
salary
department
firm
```

New Schema

```text
name
salary
department
```

The firm column is removed because it is not included in the replacement list.

---

## Why Use REPLACE COLUMNS?

Useful when:

* Multiple schema changes are required.
* Unwanted columns need to be removed.
* Column names need restructuring.
* Large schema modifications are needed.

---

# Difference Between ADD, CHANGE, and REPLACE

| Operation       | Purpose                           |
| --------------- | --------------------------------- |
| ADD COLUMNS     | Add new columns                   |
| CHANGE          | Rename or modify a column         |
| REPLACE COLUMNS | Replace complete column structure |

---

## Example Comparison

### ADD

```sql
ADD COLUMNS (company STRING)
```

Result:

```text
Existing Columns
+
company
```

---

### CHANGE

```sql
CHANGE company firm STRING
```

Result:

```text
company
    ↓
firm
```

---

### REPLACE

```sql
REPLACE COLUMNS (...)
```

Result:

```text
Entire Schema Rebuilt
```

---

# Viewing Table Structure

After any schema modification, the structure can be verified using:

```sql
DESCRIBE table_name;
```

Example:

```sql
DESCRIBE temp_exter;
```

This displays:

* Column names
* Data types
* Updated schema information

---

# Internal Role of the Metastore

All ALTER TABLE operations primarily affect the Hive Metastore.

The Metastore stores:

* Table names
* Column names
* Data types
* Storage information

When using:

```sql
ALTER TABLE
```

Hive updates metadata rather than rewriting the entire dataset.

This makes schema modifications much faster compared to recreating tables.

---

# Advantages of ALTER TABLE

### Avoids Table Recreation

No need to drop and recreate large tables.

---

### Preserves Existing Data

Data remains available during schema modifications.

---

### Supports Schema Evolution

Tables can adapt to changing business requirements.

---

### Efficient Metadata Updates

Most operations only modify metadata.

---

# Limitations

### Careful Planning Required

Improper schema changes can affect applications depending on the table.

---

### REPLACE Can Be Risky

Replacing columns incorrectly may lead to loss of schema information.

---

### Data Compatibility Issues

Changing structures without considering existing data may create inconsistencies.

---

# Quick Revision

* ALTER TABLE modifies existing table schema.
* RENAME TO changes table names.
* ADD COLUMNS adds new columns.
* CHANGE renames or modifies columns.
* REPLACE COLUMNS replaces the entire column definition.
* Most ALTER operations update metadata only.
* Existing data is generally preserved during schema changes.
* DESCRIBE can be used to verify schema modifications.

---

# One-Line Recall

**ALTER TABLE enables schema evolution in Hive by allowing table renaming, column addition, column modification, and schema replacement without recreating the table or reloading data.**
