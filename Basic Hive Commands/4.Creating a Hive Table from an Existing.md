# Creating a Hive Table from an Existing Table

## Introduction

In large-scale data processing environments, it is often necessary to create a new table based on an already existing table. Instead of defining the schema from scratch and loading data again, Hive provides mechanisms to create tables using existing tables as the source.

This is a common practice in data warehousing because it allows data engineers and analysts to create backup copies, experiment with transformations, generate reporting datasets, and work with subsets of data without affecting the original table.

---

# Why Create a Table from an Existing Table?

Consider an employee table containing millions of records.

```text
employee
│
├── emp_id
├── name
├── salary
├── department
├── joining_date
└── location
```

Directly modifying this table can be risky because:

* Production data may be affected.
* Mistakes can corrupt important datasets.
* Multiple users may be using the same table.

Instead, a copy can be created.

```text
employee
      │
      ▼
employee_copy
```

All experimentation and analysis can then be performed on the copied table.

---

# Common Use Cases

## 1. Backup Creation

Before performing major updates or transformations, a backup table can be created.

```text
employee
      │
      ▼
employee_backup
```

If something goes wrong, the original table remains intact.

---

## 2. Working with Selected Columns

A table may contain many columns that are not required for a particular analysis.

Original Table:

| emp_id | name | salary | department | address | phone |
| ------ | ---- | ------ | ---------- | ------- | ----- |

Required Table:

| emp_id | salary | department |
| ------ | ------ | ---------- |

Creating a new table with only the required columns improves readability and query performance.

---

## 3. Data Transformation

Data often passes through multiple stages.

```text
Raw Data
    │
    ▼
Staging Table
    │
    ▼
Processed Table
    │
    ▼
Reporting Table
```

Each stage may exist as a separate Hive table derived from the previous one.

---

## 4. Testing and Development

Developers frequently need a safe environment for testing.

Instead of experimenting on production data:

```text
Production Table
       │
       ▼
Test Table
```

The copied table can be modified without any risk.

---

# Source Table and Target Table

Whenever a table is created from another table, two tables are involved.

### Source Table

The existing table from which data or schema is copied.

Example:

```text
temp_ext
```

### Target Table

The newly created table.

Example:

```text
employee_copy
```

Relationship:

```text
Source Table
      │
      ▼
Target Table
```

---

# Types of Table Copying in Hive

Hive supports different forms of copying.

## Schema Only Copy

Only the structure is copied.

```text
Original Table
     │
     ▼
New Empty Table
```

Copied:

* Column names
* Data types
* Table structure

Not copied:

* Data

---

## Schema + Data Copy

Both structure and records are copied.

```text
Original Table
     │
     ▼
New Table
     │
     ▼
Same Data
```

This is the most common approach when creating backup or reporting tables.

---

# External Tables and Table Creation

An external table differs from a managed table because Hive does not own the actual data.

Hive manages only metadata.

```text
Hive Metastore
       │
       ▼
External Data Location
```

The data may reside in:

* HDFS
* Amazon S3
* Azure Data Lake
* Google Cloud Storage

or any other external storage location.

---

# Creating an External Table from an Existing External Table

When creating an external table from an existing table, the source table must itself be an external table.

Example:

```text
External Table
       │
       ▼
External Table
```

This preserves the external storage characteristics and metadata management behavior.

The new table inherits the logical structure of the source table while remaining independent.

---

# Internal Working

When Hive creates a new table from an existing table, several operations occur internally.

### Step 1: Read Metadata

Hive retrieves metadata from the Metastore.

Metadata includes:

* Column names
* Data types
* Table properties
* Storage details

---

### Step 2: Generate New Schema

A new schema is created based on the source table definition.

```text
Source Schema
      │
      ▼
Target Schema
```

---

### Step 3: Create Storage Location

Hive allocates storage for the new table.

Depending on table type:

* Managed Table → Hive Warehouse
* External Table → External Location

---

### Step 4: Copy Data

Data records are copied from source to target.

```text
Source Data
      │
      ▼
Target Data
```

---

### Step 5: Register in Metastore

The new table is registered in the Hive Metastore and becomes available for querying.

---

# Benefits of Creating Tables from Existing Tables

## Reduced Development Effort

No need to redefine complex schemas manually.

---

## Consistency

The copied table follows the same schema as the source table.

---

## Faster Data Preparation

Large datasets can be reused quickly.

---

## Safe Experimentation

Users can test queries and transformations without touching production data.

---

## Better Data Organization

Different versions of datasets can be maintained for different purposes.

---

# Real-World Example

Consider a company storing employee data.

```text
employee_master
```

The HR team requires all employee details.

However, the Finance team only needs:

```text
employee_salary
```

Instead of repeatedly querying the master table, a separate table can be created.

```text
employee_master
        │
        ▼
employee_salary
```

This improves performance and simplifies reporting.

---

# Important Points

* Existing tables can be used as templates for creating new tables.
* New tables may contain only structure or both structure and data.
* External tables are commonly used when data exists outside Hive Warehouse.
* Creating copies is useful for backups, reporting, transformations, and testing.
* Metadata is always managed through the Hive Metastore.
