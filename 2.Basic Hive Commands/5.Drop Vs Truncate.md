# DROP TABLE and TRUNCATE TABLE in Hive

## Introduction

Managing tables in Hive involves more than just creating and loading data. In real-world environments, tables often need to be deleted, cleaned, recreated, or refreshed. Hive provides two important commands for this purpose:

1. **DROP TABLE** – Removes a table from Hive.
2. **TRUNCATE TABLE** – Removes all records from a table while keeping the table structure intact.

Although both commands affect data storage, they behave differently depending on whether the table is a **Managed Table** or an **External Table**. Understanding this difference is extremely important because an incorrect command can result in unintended data loss.

---

# DROP TABLE Command

## Purpose

The DROP TABLE command is used to remove a table from Hive.

Syntax:

```sql
DROP TABLE table_name;
```

Example:

```sql
DROP TABLE employee;
```

When this command is executed, Hive removes the table from the Metastore and performs additional actions depending on the table type.

---

# What Happens When a Managed Table is Dropped?

Managed tables are completely controlled by Hive.

Hive owns:

* Metadata
* Table directory
* Data files

Structure:

```text
Hive
 │
 ├── Metadata
 │
 └── Data Files
```

Suppose the table is stored as:

```text
/user/hive/warehouse/demo_01.db/employee
```

When the table is dropped:

```sql
DROP TABLE employee;
```

Hive performs two operations:

### Step 1: Remove Metadata

Deletes table information from the Metastore.

```text
Table Name
Column Names
Data Types
Properties
```

are removed.

### Step 2: Remove Data Files

Deletes the entire table directory from HDFS.

```text
Before Drop

employee
   │
   └── employees.txt
```

```text
After Drop

employee directory removed
```

Result:

```text
Metadata Deleted
+
Data Deleted
```

Everything is permanently removed.

---

# What Happens When an External Table is Dropped?

External tables work differently.

Hive manages only the metadata.

The data exists outside Hive's warehouse directory.

Structure:

```text
Hive Metastore
       │
       ▼
External Data Location
```

Example:

```text
/hive_external_warehouse/employees.txt
```

When:

```sql
DROP TABLE employee_ext;
```

is executed:

### Step 1: Metadata Removed

Hive removes:

* Table definition
* Column information
* Table properties

from the Metastore.

### Step 2: Data Remains

The actual file remains untouched.

```text
Before Drop

employee_ext
      │
      ▼
employees.txt
```

```text
After Drop

Metadata Removed

employees.txt still exists
```

Result:

```text
Metadata Deleted
+
Data Preserved
```

---

# Why External Tables Are Important

External tables are useful when multiple systems share the same data.

Example:

```text
Spark
  │
Hive
  │
Kafka
  │
Data Lake
```

If Hive deleted the actual files whenever a table was dropped, other systems would lose access to the data.

External tables prevent this problem.

---

# DROP TABLE Summary

| Table Type     | Metadata | Data      |
| -------------- | -------- | --------- |
| Managed Table  | Deleted  | Deleted   |
| External Table | Deleted  | Preserved |

This is one of the most frequently asked Hive interview questions.

---

# TRUNCATE TABLE Command

## Purpose

TRUNCATE TABLE removes all records from a table while preserving the table definition.

Syntax:

```sql
TRUNCATE TABLE table_name;
```

Example:

```sql
TRUNCATE TABLE employee;
```

---

# Difference Between DROP and TRUNCATE

### DROP TABLE

Removes:

* Table
* Metadata
* Data

Result:

```text
Table No Longer Exists
```

---

### TRUNCATE TABLE

Removes:

* Data Only

Keeps:

* Table
* Schema
* Metadata

Result:

```text
Empty Table Exists
```

---

# Internal Working of TRUNCATE

Consider:

```text
employee
 │
 └── employees.txt
```

When:

```sql
TRUNCATE TABLE employee;
```

is executed:

### Metadata Remains

Hive still knows:

```text
employee
emp_id
name
salary
```

All schema information remains unchanged.

---

### Data Is Removed

The records inside the table are deleted.

Before:

```text
employee
 │
 └── employees.txt
       ├── Record 1
       ├── Record 2
       ├── Record 3
       └── Record 4
```

After:

```text
employee
 │
 └── Empty
```

---

### Table Directory Remains

The directory still exists.

Before:

```text
warehouse
   │
   └── employee
          │
          └── employees.txt
```

After:

```text
warehouse
   │
   └── employee
```

Directory remains but contains no data.

---

# Why Use TRUNCATE?

TRUNCATE is useful when:

### Reloading Data

```text
Old Data
   ↓
TRUNCATE
   ↓
Load New Data
```

---

### Clearing Temporary Tables

Temporary processing tables often need to be emptied regularly.

---

### Faster Than DROP + CREATE

Instead of:

```sql
DROP TABLE employee;

CREATE TABLE employee(...);
```

we simply use:

```sql
TRUNCATE TABLE employee;
```

which preserves the schema.

---

# Why TRUNCATE Works Only on Managed Tables

Hive allows:

```sql
TRUNCATE TABLE managed_table;
```

But does not allow:

```sql
TRUNCATE TABLE external_table;
```

---

## Reason

Hive owns managed table data.

Therefore Hive can safely remove it.

```text
Hive
 │
 ├── Metadata
 └── Data
```

---

For external tables:

```text
Hive
 │
 └── Metadata

Data
 │
 └── External Location
```

Hive does not own the external data.

Deleting that data could affect:

* Spark jobs
* ETL pipelines
* Data lakes
* Other applications

Therefore Hive prevents truncation.

---

# Error When Truncating External Tables

Attempting:

```sql
TRUNCATE TABLE employee_ext;
```

typically produces an error similar to:

```text
Cannot truncate non-managed table
```

This protection mechanism prevents accidental deletion of externally managed data.

---

# DROP vs TRUNCATE

| Feature                    | DROP TABLE | TRUNCATE TABLE |
| -------------------------- | ---------- | -------------- |
| Removes Data               | Yes        | Yes            |
| Removes Metadata           | Yes        | No             |
| Removes Table Structure    | Yes        | No             |
| Table Exists After Command | No         | Yes            |
| Works on Managed Tables    | Yes        | Yes            |
| Works on External Tables   | Yes        | No             |
| Recoverability             | Difficult  | Schema Remains |

---

# Real-World Example

Consider a sales table loaded daily.

```text
sales_daily
```

Every morning:

```sql
TRUNCATE TABLE sales_daily;
```

New data is then loaded.

This avoids recreating the table every day.

---

For an external data lake table:

```sql
DROP TABLE customer_ext;
```

The table disappears from Hive, but the data files remain available for Spark, Databricks, or other tools.

---

# Interview Questions

### What happens when a managed table is dropped?

Both metadata and data are deleted.

---

### What happens when an external table is dropped?

Only metadata is deleted. Data remains in the external location.

---

### What is the purpose of TRUNCATE TABLE?

To remove all records while preserving schema and metadata.

---

### Why can't external tables be truncated?

Because Hive does not own the underlying data and therefore cannot safely delete it.

---

### Difference between DROP and TRUNCATE?

DROP removes the entire table, while TRUNCATE removes only the data.

---

# Quick Revision

* `DROP TABLE` removes a table from Hive.
* Managed Table → Metadata + Data deleted.
* External Table → Metadata deleted, Data preserved.
* `TRUNCATE TABLE` removes all records but keeps schema.
* Table structure remains after truncation.
* Hive allows truncation only on managed tables.
* External tables cannot be truncated because Hive does not own their data.
* Managed vs External behavior is a very common interview topic.

---

# One-Line Recall

**DROP TABLE removes the table itself, while TRUNCATE TABLE only removes its data; managed tables lose both metadata and data on drop, whereas external tables lose only metadata.**

# 5-Mark Exam Answer

**DROP TABLE and TRUNCATE TABLE are two important Hive commands. DROP TABLE removes the table metadata and data. For managed tables, both data and metadata are deleted, whereas for external tables only metadata is removed and data remains in the external location. TRUNCATE TABLE removes all records from a managed table while preserving its metadata and structure. Hive does not allow truncating external tables.**
