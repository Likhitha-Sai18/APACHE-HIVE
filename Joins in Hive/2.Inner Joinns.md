# Joins in Hive – Practical Example with Coding Platform Dataset

This section introduces joins using a real-world coding platform dataset. Instead of learning joins using abstract tables, the dataset models how platforms like LeetCode, HackerRank, or CodeChef store information.

---

# Dataset Overview

The dataset consists of three tables:

```text
Users
   │
   ├── User Information
   │
Problems
   │
   ├── Problem Information
   │
Submissions
   │
   ├── User Attempts
```

These tables are related to each other through common columns.

---

# Users Table

Stores information about registered users.

| Column  | Description                     |
| ------- | ------------------------------- |
| user_id | Unique identifier for each user |
| name    | User name                       |
| country | User country                    |
| rating  | User rating                     |

Example:

| user_id | name  | country | rating |
| ------- | ----- | ------- | ------ |
| 101     | John  | India   | 1800   |
| 102     | Alice | USA     | 2200   |
| 103     | Bob   | India   | 1600   |

---

# Problems Table

Stores information about coding problems.

| Column     | Description       |
| ---------- | ----------------- |
| problem_id | Unique problem ID |
| difficulty | Difficulty level  |
| points     | Reward points     |
| tag        | Problem category  |

Example:

| problem_id | difficulty | points | tag    |
| ---------- | ---------- | ------ | ------ |
| P101       | Easy       | 50     | Arrays |
| P102       | Medium     | 100    | Greedy |
| P103       | Hard       | 200    | Graph  |

---

# Submissions Table

Stores successful submissions.

| Column     | Description        |
| ---------- | ------------------ |
| user_id    | User who submitted |
| problem_id | Submitted problem  |
| attempts   | Number of attempts |

Example:

| user_id | problem_id | attempts |
| ------- | ---------- | -------- |
| 101     | P101       | 2        |
| 102     | P102       | 1        |
| 101     | P103       | 4        |

---

# Relationship Between Tables

```text
Users
  │
  │ user_id
  ▼
Submissions
  ▲
  │ problem_id
  │
Problems
```

### Join Keys

Between Users and Submissions:

```text
user_id
```

Between Problems and Submissions:

```text
problem_id
```

These common columns are used for joins.

# Business Question

The instructor wants to answer:

> Which countries have the highest number of unique users who successfully submitted solutions?

To answer this:

We need:

### Users Table

Contains:

```text
country
```

### Submissions Table

Contains:

```text
user_id
```

Neither table alone contains enough information.

Therefore:

```text
Users
   +
Submissions
```

must be joined.

---

# Why Inner Join?

We only want users who actually submitted solutions.

Some users may register but never submit.

Example:

### Users

| user_id | country |
| ------- | ------- |
| 101     | India   |
| 102     | USA     |
| 103     | India   |
| 104     | UK      |

---

### Submissions

| user_id |
| ------- |
| 101     |
| 102     |
| 103     |

Notice:

```text
104
```

has never submitted.

An INNER JOIN automatically removes such users.

---

# Query Breakdown

## Step 1: Join Users and Submissions

```sql
SELECT *
FROM users u
INNER JOIN submissions s
ON u.user_id = s.user_id;
```

---

### Output

| user_id | country | problem_id |
| ------- | ------- | ---------- |
| 101     | India   | P101       |
| 102     | USA     | P102       |
| 103     | India   | P103       |

User 104 disappears because no submission exists.

---

# Step 2: Count Users Per Country

```sql
SELECT
country,
COUNT(DISTINCT u.user_id)
FROM users u
INNER JOIN submissions s
ON u.user_id = s.user_id
GROUP BY country;
```

---

## Why DISTINCT?

Suppose:

### Submissions

| user_id |
| ------- |
| 101     |
| 101     |
| 101     |
| 102     |

Without DISTINCT:

```text
India = 3
```

which is incorrect.

We want:

```text
India = 1 User
```

Therefore:

```sql
COUNT(DISTINCT user_id)
```

is used.

---

# Step 3: Sort Results

```sql
ORDER BY COUNT(DISTINCT u.user_id) DESC;
```

Purpose:

Show countries with highest participation first.

---

# Complete Query

```sql
SELECT
    country,
    COUNT(DISTINCT u.user_id) AS total_users
FROM users u
INNER JOIN submissions s
ON u.user_id = s.user_id
GROUP BY country
ORDER BY total_users DESC;
```

---

# Example Output

| Country | Users |
| ------- | ----- |
| India   | 2500  |
| USA     | 1800  |
| UK      | 900   |
| Germany | 600   |

This tells us which countries contribute the most active users.

---

# Why Multiple MapReduce Jobs Were Created?

The instructor observed:

```text
3 MapReduce Jobs
```

for a single query.

Many beginners assume:

```text
1 Query = 1 Job
```

This is not always true.

---

## Job 1 – Join

```sql
INNER JOIN
```

Hive must combine two datasets.

```text
Users
   +
Submissions
```

This requires a MapReduce phase.

---

## Job 2 – Group By

```sql
GROUP BY country
```

Records must be grouped.

Example:

```text
India
India
USA
India
USA
```

becomes

```text
India → 3
USA → 2
```

This requires another MapReduce operation.

---

## Job 3 – Order By

```sql
ORDER BY total_users
```

Results must be sorted.

Sorting large datasets requires another MapReduce stage.

---

# Query Execution Flow

```text
Users Table
      │
      ▼
 INNER JOIN
      │
      ▼
 Joined Data
      │
      ▼
 GROUP BY Country
      │
      ▼
 Country Counts
      │
      ▼
 ORDER BY Count
      │
      ▼
 Final Output
```

# Interview Questions

### Why was INNER JOIN used?

Because only users who submitted solutions should be counted.

---

### What is the join column?

```text
user_id
```

---

### Why use COUNT(DISTINCT)?

To avoid counting the same user multiple times.

---

### Why use GROUP BY?

To aggregate users country-wise.

---

### Why use ORDER BY?

To sort countries based on user count.

---

### Why did Hive create multiple MapReduce jobs?

Because JOIN, GROUP BY, and ORDER BY each require separate processing stages.

# Quick Revision

* Users table contains user information.
* Problems table contains coding problem information.
* Submissions table links users and problems.
* `user_id` joins Users and Submissions.
* `problem_id` joins Problems and Submissions.
* INNER JOIN returns only users who submitted solutions.
* GROUP BY aggregates country-wise statistics.
* COUNT(DISTINCT) prevents duplicate counting.
* ORDER BY sorts final results.
* JOIN + GROUP BY + ORDER BY may generate multiple MapReduce jobs.
