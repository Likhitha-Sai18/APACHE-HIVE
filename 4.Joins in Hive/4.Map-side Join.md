# Map-Side Join in Hive

## Introduction

Joins are among the most expensive operations in Hive because they involve combining data from multiple tables. In a distributed environment such as Hadoop, data is stored across multiple machines. When a join query is executed, Hive must bring together matching records from different tables before producing the final result.

In a normal Hive join, large amounts of data are transferred across the cluster and processed in the reducer phase. This data movement, known as **shuffle**, is often the most expensive part of query execution. To reduce this overhead, Hive provides an optimization technique called **Map-Side Join**.

Map-Side Join allows Hive to perform the join operation during the map phase itself, avoiding the need for reducers and significantly improving performance.

---

# Why Normal Joins Are Expensive

Consider two tables:

### Users Table

| user_id | name  |
| ------- | ----- |
| 1       | John  |
| 2       | Alice |
| 3       | Bob   |

Size: 80 KB

### Submissions Table

| user_id | problem_id |
| ------- | ---------- |
| 1       | P101       |
| 1       | P102       |
| 2       | P201       |
| 3       | P301       |

Size: 3.2 MB

Suppose we execute:

```sql
SELECT *
FROM users u
JOIN submissions s
ON u.user_id = s.user_id;
```

The objective is to combine user information with submission information.

In a traditional Hive join, the process occurs as follows:

### Step 1: Table Scan

The mapper reads both tables separately.

```text
Users Table
     ↓
   Mapper

Submissions Table
     ↓
   Mapper
```

At this stage, Hive has only read the data. No join has occurred yet.

---

### Step 2: Shuffle and Sort

To perform the join, all records having the same join key (`user_id`) must be sent to the same reducer.

For example:

```text
Users:
1 John

Submissions:
1 P101
1 P102
```

These records may initially reside on different machines.

Hive therefore transfers them across the network so that they arrive at the same reducer.

This process is called:

```text
Shuffle and Sort
```

During this stage:

* Data is moved between nodes.
* Join keys are sorted.
* Records with the same key are grouped together.

This stage consumes:

* Network bandwidth
* CPU resources
* Disk I/O

For very large datasets, the shuffle phase becomes the major bottleneck.

---

### Step 3: Reduce Phase

Once matching records arrive at the same reducer, Hive performs the join.

```text
Reducer

Users:
1 John

Submissions:
1 P101
1 P102

↓ Join

1 John P101
1 John P102
```

The final joined result is then returned.

---

# Problem with Reduce-Side Joins

Imagine the following scenario:

```text
Users Table       → 5 GB
Submissions Table → 500 GB
```

To perform the join, Hive must:

1. Read 505 GB of data.
2. Shuffle a large amount of data across the cluster.
3. Sort records.
4. Execute reducers.

Even if one table is very small, Hive still performs the expensive shuffle process.

This is inefficient.

---

# The Idea Behind Map-Side Join

Hive developers observed that many joins involve one large table and one very small table.

Example:

```text
Countries Table → 5 KB
Customers Table → 300 GB
```

Instead of moving both tables across the network, why not load the small table into memory and use it while scanning the large table?

This idea leads to Map-Side Join.

---

# What is a Map-Side Join?

A Map-Side Join is an optimization technique where Hive loads the smaller table into memory and performs the join directly during the map phase.

Because the join is completed within the mapper, there is no need for:

* Shuffle
* Sort
* Reducer

This makes the query significantly faster.

---

# Internal Working of Map-Side Join

## Step 1: Identify the Smaller Table

Hive first determines which table is smaller.

Example:

```text
Users Table       → 80 KB
Submissions Table → 3.2 MB
```

The Users table is chosen because it is much smaller.

---

## Step 2: Load Smaller Table into Memory

The entire Users table is loaded into memory.

```text
Memory Cache

1 → John
2 → Alice
3 → Bob
```

Internally, Hive usually stores this information as a hash table.

```text
user_id
   ↓
User Record
```

Example:

```text
{
 1 → John,
 2 → Alice,
 3 → Bob
}
```

This allows very fast lookups.

---

## Step 3: Scan the Larger Table

Hive now scans the larger table.

```text
Submissions

1 P101
1 P102
2 P201
3 P301
```

Each row is read by the mapper.

---

## Step 4: Perform Join in Mapper

As each submission record is read, Hive immediately looks up the corresponding user information from memory.

Example:

Reading:

```text
1 P101
```

Lookup:

```text
1 → John
```

Output:

```text
1 John P101
```

Reading:

```text
2 P201
```

Lookup:

```text
2 → Alice
```

Output:

```text
2 Alice P201
```

The join is completed instantly.

No reducer is required.

---

# Execution Flow Comparison

## Reduce-Side Join

```text
Users
   ↓
 Mapper

Submissions
   ↓
 Mapper

   ↓
 Shuffle

   ↓
 Sort

   ↓
 Reducer

   ↓
 Join

   ↓
 Output
```

---

## Map-Side Join

```text
Users (Small)
      ↓
 Load into Memory

Submissions (Large)
      ↓
 Mapper
      ↓
 Join
      ↓
 Output
```

Notice how the entire shuffle and reduce stages disappear.

---

# Why Map-Side Join Is Faster

## No Shuffle

Data does not need to be transferred across machines.

This saves significant network overhead.

---

## No Sorting

Records do not need to be sorted before joining.

Sorting is one of the most expensive operations in distributed systems.

---

## No Reducers

The reducer phase is completely eliminated.

This reduces execution time and cluster resource consumption.

---

## Fast Memory Access

The small table is stored in RAM.

Accessing RAM is dramatically faster than reading from disk or transferring data over a network.

---

# Enabling Map-Side Join

Hive can automatically convert suitable joins into Map-Side Joins.

Enable this behavior using:

```sql
SET hive.auto.convert.join=true;
```

After enabling this setting, Hive automatically checks table sizes.

If one table is sufficiently small, Hive converts the query into a Map-Side Join.

---

# Using EXPLAIN to Verify

The EXPLAIN command shows Hive's execution plan.

```sql
EXPLAIN
SELECT *
FROM users u
JOIN submissions s
ON u.user_id = s.user_id;
```

When Map-Side Join is not used:

```text
Map
 ↓
 Reduce
 ↓
 Join
```

When Map-Side Join is selected:

```text
Map
 ↓
 Join
```

The absence of a reducer stage indicates that Hive is performing a Map-Side Join.

---

# When Map-Side Join Works Best

Map-Side Join is ideal when:

### One Table Is Very Small

```text
Countries Table → 5 KB
Customers Table → 300 GB
```

### Lookup Tables

Examples:

* Country codes
* Product categories
* Department information
* Region mappings

These tables are typically small enough to fit into memory.

---

# When Map-Side Join Should Not Be Used

Suppose:

```text
Table A → 100 GB
Table B → 120 GB
```

Neither table can fit into memory.

In this situation:

```text
Map-Side Join ❌
```

Hive must use:

```text
Reduce-Side Join ✔
```

because loading 100 GB into memory is not practical.

---

# Real-World Example

Consider an e-commerce company.

### Products Table

| product_id | category    |
| ---------- | ----------- |
| P101       | Electronics |
| P102       | Clothing    |

Size:

```text
20 KB
```

### Orders Table

Contains hundreds of millions of records.

Size:

```text
500 GB
```

Query:

```sql
SELECT *
FROM orders o
JOIN products p
ON o.product_id = p.product_id;
```

Hive loads the small Products table into memory and streams the large Orders table through mappers.

This is an ideal Map-Side Join scenario.

---

# Advantages of Map-Side Join

* Eliminates shuffle phase.
* Eliminates sorting.
* Eliminates reducers.
* Reduces network traffic.
* Improves query performance.
* Uses memory efficiently for small lookup tables.
* Particularly useful in data warehousing workloads.

---

# Limitations

* Requires at least one table to be small.
* Memory limitations restrict usage.
* Not suitable for joins involving multiple large tables.
* Excessively large "small" tables can cause memory pressure.

# Interview Points

### What is a Map-Side Join?

A Hive optimization where the smaller table is loaded into memory and the join is executed during the map phase.

### Why is it faster?

Because it avoids shuffle, sorting, and reducer execution.

### Which table is loaded into memory?

The smaller table.

### How do you enable automatic Map-Side Join?

```sql
SET hive.auto.convert.join=true;
```

### How can you verify if Hive is using Map-Side Join?

Use:

```sql
EXPLAIN query;
```

and check whether the reducer stage is absent.

# Quick Revision

* Normal Hive joins perform joins in reducers.
* Reducers require shuffle and sorting.
* Shuffle is expensive.
* Map-Side Join loads the smaller table into memory.
* Join happens inside the mapper.
* No shuffle and no reducer are required.
* Enable automatic conversion using `hive.auto.convert.join=true`.
* Best suited for large-table-to-small-table joins.

# One-Line Recall

**Map-Side Join improves Hive performance by loading a small table into memory and performing the join directly in the mapper, eliminating expensive shuffle and reduce operations.**
